# 决策与设计（指南第一、二部分）

## §1 三判据闸门（Baseline 先行）

拿 3–5 个**真实**场景，不开 skill 让 Agent 裸跑，记录观察后填表：

| 判据 | 满足？ |
|---|---|
| 同一类问题重复出现（≥3 次） | ☐ |
| Agent 每次跑偏，或每次要重新解释同一套口径 | ☐ |
| 有明确可验收的交付物（文件/代码/报告/配置） | ☐ |

**三条全满足才开始设计。** 最容易失败的开局是"万能助手 Skill"；最稳的起点是
"重复 + 易跑偏 + 有交付物"的小而特化场景。

**上下文税**：每个 skill 的 Level 1 元数据常驻上下文（20 个 skill ≈ 1000–3000
token），边界模糊、少触发、没人维护的 skill 污染路由。定期清理。

## §2 选骨架（4 选 1，决定内容组织）

```
主要做什么？
├─ 有顺序的操作（部署/迁移/转换）      → A Workflow 流程型：Overview→Steps
├─ 同主题多种独立操作（PDF/数据工具）   → B Task 任务菜单型：Quick Start→Tasks
├─ 输出规范/标准（代码规范/评审标准）   → C Reference 规范型：原则→规则表→反例
└─ 一组独立能力按需调用（综合系统）     → D Capabilities 能力清单型
```

先选骨架再写内容——比对着空白页更有方向，结构也一致。

## §2.2 叠加工作流模式（6+1，决定执行控制）

| 模式 | 适用 | 关键技巧 |
|---|---|---|
| 1 线性流程 | 明确步骤的操作（骨架 A 默认） | 安全默认值、每步具体命令、超时提示、降级方案、负面指令 |
| 2 决策树+按需加载 | 10+ 分支的大知识域 | 用用户语言分类（"I need to run code"）、主文件小 references 大、可拆导航型+操作型两个 skill |
| 3 循环迭代 | 单会话内反复"做→验证→改进"（TDD） | 铁律、强硬语气、借口反驳表、退出条件清单 |
| 4 接力棒循环 | 跨 session 长期项目 | 文件即状态（next-prompt.md 接力棒）、"写下一棒"标 CRITICAL、编排无关 |
| 5 多阶段+检查点 | 跨天/周、有 Go/No-Go 决策点 | 统一 Phase 模板（Activities→Outputs→Decision Point）、NO 路径标时间成本、主 skill 编排子 skill |
| 特 思维框架 | 控制"怎么想"而非"做什么"（审计/深度分析） | 量化阈值（≥3 不变量/≥5 假设）、非目标约束（先理解禁判断）、反幻觉规则 |

模式 3 vs 4：状态存对话上下文 vs 外部文件；单会话 vs 跨 session；checklist 打勾
退出 vs 路线图清空退出。

## §2.3 粒度：SKILL.md 是路由器，不是知识仓库

单一职责：一个 skill 只做一件事。SKILL.md 只留意图路由表 + 全局安全红线，业务
细节下沉模块文件。

**拆分信号**（任一命中即考虑拆）：

| 信号 | 操作 |
|---|---|
| 正文超 300 行 / 某 Step 超 100 行 | 细节下沉子模块 |
| 文件超 500 行（硬上限） | 必须拆 |
| 含多个可独立使用的工作流 | 拆成子 skill |
| 不同部分改动频率差异大 | 拆开分别维护 |
| 多个 skill 有重复前置步骤 | 提取通用前置 skill |

**引用子文档必须写契约**（不能只给路径）：

```
读取并执行 [环境初始化](steps/00-environment-setup.md)。
触发条件：项目首次运行时
预期产出：环境检查通过，输出 {"status": "ready"}
检查点：bash scripts/check-env.sh
```
