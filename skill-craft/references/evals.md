# Evals：先写评测，再写 Skill（指南第五部分）

## 核心理念

**评测不是事后验证，而是约束先行。** Skill 是"让评测用例通过"的最小实现。

| 传统顺序 | 评测驱动顺序 |
|---|---|
| 写 skill → 测试 → 修改 | 写评测 → 写最小 skill → 迭代 |
| 事后发现问题 | 边界约束先于实现存在 |
| 改完不知有没有改坏 | iteration-N 对比防回归 |

## 六步流程

1. **Baseline**：无 skill 跑 3–5 真实 case，记录哪里不稳/走偏/误用。
2. **先写评测**：evals.json 定义"什么算通过/失败"。
3. **最小 skill**：只覆盖能通过评测的最小规则集，避免早期过度复杂化。
4. **补边界**：边界条件 + Good/Bad 结构化示例。
5. **回归验证**：任何改动必须通过已有评测。
6. **真实校准**：真实使用发现新问题 → 回到 2，下一轮。

## 目录结构

```
my-skill/evals/
├── evals.json       ← 用例集（持续追加）
├── iteration-1/     ← 每次迭代产出，不覆盖旧版
│   ├── results.json
│   └── benchmark.md
└── iteration-2/…
```

## evals.json 模板

```json
{
  "skill_name": "your-skill-name",
  "version": "1.0",
  "evals": [
    { "id": "core-1", "category": "core",
      "note": "最常见的标准请求，用口语写",
      "prompt": "[像真实用户说话]",
      "expected_output": "[应做到什么]",
      "assertions": [
        { "id": "a1", "text": "完成了[主要动作]" },
        { "id": "a2", "text": "提供了[验证步骤]" },
        { "id": "a3", "text": "没有[禁止行为]" } ] },
    { "id": "core-2", "category": "core", "note": "同一任务口语化变体", "...": "..." },
    { "id": "edge-1", "category": "edge",
      "note": "边界：输入不完整/已完成/不需要操作",
      "expected_output": "识别后停下或告知，不强行执行",
      "assertions": [
        { "id": "e1", "text": "运行了前置检查" },
        { "id": "e2", "text": "没有强行执行" } ] },
    { "id": "edge-2", "category": "edge", "note": "边界：相关但不该触发", "...": "..." },
    { "id": "gotcha-1", "category": "gotcha",
      "note": "已知坑：只写真实发生过的失败",
      "assertions": [ { "id": "g1", "text": "没有重复[历史错误行为]" } ] }
  ]
}
```

## 三类用例要点

| 类型 | 数量 | 要点 | 易犯错 |
|---|---|---|---|
| core | 5–10 | 同任务 2–3 种说法变体；真实用户语言 | 只写 1 个，覆盖不够 |
| edge | 3–5 | 空输入/信息不全/相关但不该触发/超范围 | 只测 happy path |
| gotcha | 3–5 | 只写真实发生的失败，每发现新坑追加 | 凭感觉猜坑 |

> 边界和已知坑才是真正暴露问题的地方，别只跑 core。

## Gotcha 飞轮

**遇到问题不要重写 skill——把失败追加为 gotcha。** 条目类型：不该触发的相似输入 /
易被误用的工具 / 该停下等确认的边界输入 / 连续失败 N 次的降级路径。skill 越成熟
不一定越长，但更清楚"哪些路不能走"。

## iteration-N 对比

每次改 skill → `evals/iteration-N+1/` 重跑，**不覆盖旧结果**。确认：目标问题改善
（通过率↑）+ 没引入新问题（其他用例无退步）。规则变严→查误伤；变宽→查漏触发。

## 质量达标线

| 指标 | 达标 | 优秀 |
|---|---|---|
| 触发准确率 Precision | ≥85% | ≥95% |
| 触发召回率 Recall | ≥85% | ≥95% |
| 效果通过率 | ≥80% | ≥90% |
| 相对提升率（vs 无 skill） | ≥30% | ≥50% |

触发实测的两级做法：① 面板法——评委面前摆本 skill + 同域近邻 + 真实干扰技能的
description 原文做多数票路由；② 真环境法——带完整已装技能清单的子 agent 或
`claude -p` 实测。面板 100% 不代表真环境不丢单，边界问法务必进真环境验。
