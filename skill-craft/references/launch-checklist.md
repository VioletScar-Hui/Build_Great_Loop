# 上线前检查（指南第六部分 + 反模式 + 四维自测）

> 逐项打勾后再上线；任一红项先修再上。

## ✦ Description（Level 1）

- [ ] 同时回答 WHAT 和 WHEN
- [ ] 枚举 ≥3 个具体触发词/触发句，含口语化说法
- [ ] 第三人称（非"我可以帮你…"）
- [ ] 有排除边界（"不适用于…"）
- [ ] ≤1024 字符
- [ ] 通过 20 问测试（10 该触发 + 10 不该触发，≥85%）
- [ ] 与同类 skill 无 description 语义重叠

## ✦ 正文（Level 2）

- [ ] 开头说清 做什么/为什么/何时不做
- [ ] 指令全是祈使句
- [ ] 重要规则解释了"为什么"，非只堆 MUST
- [ ] 有 Before/After 或 ≥3 个 Few-Shot
- [ ] 关键步骤有可直接运行的检查点命令
- [ ] 正文 ≤500 行
- [ ] 易偷懒步骤有借口反驳表或负面指令
- [ ] 无硬编码数值（超时/缓冲区），用判断规则

## ✦ 结构（Level 3）

- [ ] 长规范/API 文档已下沉 references/，引用写了契约
- [ ] 确定性计算封装 scripts/（不让 LLM 算）
- [ ] 脚本符合四原则（自愈/结构化输出/幂等/安全边界）
- [ ] examples/ 有 ≥1 个完整优质样例

## ✦ Evals

- [ ] evals.json：core ≥5、edge ≥1
- [ ] 有/无 skill 各跑一次，相对提升 ≥30%
- [ ] 所有 assertions 可判定通过/失败（非主观感受）
- [ ] 用例 prompt 是真实用户语言

## 反模式 11 条速查

| # | 反模式 | 症状 | 解法 |
|---|---|---|---|
| 1 | 大杂烩 | 一个 skill 干三件事 | 拆分，单一职责 |
| 2 | Description 黑话 | 全是内部术语 | 通用语言 + 具体关键词 |
| 3 | 没有示例 | 纯文字零示例 | 加 Few-Shot / Before-After |
| 4 | 没有验证点 | 做完才检查，错误被放大 | 关键步骤加检查点 |
| 5 | 写死数值 | 硬编码"超时 30 秒" | 给判断规则和参考范围 |
| 6 | 当 Wiki 写 | 背景 300 行指令 50 行 | 背景下沉，正文只留"做什么" |
| 7 | 无 Evals | 改了不知道有没有改坏 | 维护 evals/，每改必回归 |
| 8 | Skill = Prompt | 一次性对话提示，缺结构约束 | 写成"指令文件"，下达指令而非解释原理 |
| 9 | 越复杂越强大 | 职责混乱命中率反降 | 单一职责边界清晰 |
| 10 | 全堆一个文件 | 超长 SKILL.md 噪声带偏 | 渐进披露：下沉 references/scripts |
| 11 | 脚本魔法数字 | `TIMEOUT=30` 无解释 | 语义化命名 + 来源注释 + 可参数覆盖 |

## 四维自测（5 分钟）

| 维度 | 自测问题 | 好 ✅ | 坏 ❌ |
|---|---|---|---|
| 单一职责 | 一句话说得清做什么吗 | 能且清楚 | 要说好几件事 |
| 描述清晰度 | 陌生人读 description 知道何时用吗 | 马上知道 | 要解释才懂 |
| 可测试性 | 写得出 assertions 吗 | 判断标准明确 | "要看情况" |
| 可组合性 | 输出能被其他 skill 直接用吗 | 格式明确可传递 | 依赖上下文理解 |

**四项全绿 → 上线。任一红 → 先修。**
