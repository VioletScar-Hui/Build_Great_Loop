## 一、压力场景测试法（TDD for skills 的强化形态）

**铁律：NO SKILL WITHOUT A FAILING TEST FIRST**——对新建和修改同样适用。这与
指南的"评测先行"同源，但把"先看失败"做成了硬流程：

- **RED（基线）**：不带 skill，让子 agent 跑压力场景，**逐字记录**它的选择和
  借口原话。没看过裸跑失败，就不知道 skill 该教什么。
- **GREEN（最小 skill）**：只针对基线暴露的具体借口写规则，不为假想情况加料。
  带 skill 重跑同场景，确认合规。
- **REFACTOR（堵洞）**：agent 找到新借口 → 加显式反驳 → 重测，直到防弹。

**压力类型**（纪律型 skill 要组合 ≥3 种）：时间压力（"赶时间"）、沉没成本
（"都写这么多了"）、权威（"老板说跳过"）、疲惫（长会话末尾）。

**四类 skill 的分型测试**：

| 类型 | 测什么 | 通过标准 |
|---|---|---|
| 纪律型（TDD/验证先行） | 组合压力下是否守规 | 最大压力下仍守规 |
| 技法型（how-to） | 新场景能否正确套用 + 指令有无缺口 | 成功应用于新场景 |
| 模式型（思维模型） | 能否识别 何时适用/何时不适用 | 含反例判断正确 |
| 参考型（API/文档） | 能否检索到 + 用对 | 常见用例全覆盖 |

**防弹三件套**（纪律型必备，与指南的借口反驳表叠加）：
1. 逐洞显式封堵——不只说规则，还禁掉每个具体绕法（"不许留作参考、不许边跑边
   改、删除就是删除"）；
2. 早置一句"**违反字面就是违反精神**"，掐死整类"我遵守的是精神"式借口；
3. **Red Flags 自查清单**——把"我正要犯规"的症状列出来（"代码在测试前"、
   "我手测过了"、"这次不一样因为…"→ 全部意味着停下重来）。

## 二、CSO（Claude Search Optimization）——description 的实测陷阱

**核心发现（实测）**：description 里**摘要工作流**会让 Claude 照着 description
的摘要走捷径、跳过正文——写着"任务间做代码评审"就只评审一次，哪怕正文流程图
明明是两次。改成只写触发条件后，Claude 才去读正文。

**规则**：description 只回答"什么时候该读我"，**不要复述步骤/箭头流程**。
与指南 §3.1 公式的融合法：功能定义（WHAT）+触发词（WHEN）+排除边界照写，
但"核心能力"写成能力名词清单，不写 A→B→C 的流程箭头。

**其余 CSO 要点**：
- 关键词覆盖：报错原文、症状词（flaky/hanging）、同义词、工具名——写用户会
  说的词；
- 命名动词优先（creating-skills > skill-creation）；-ing 适合流程类；
- 跨 skill 引用只写名字 + 是否必读标记，**不用 @ 路径**（@ 会立刻强制加载、
  烧上下文）；
- token 预算：高频加载的 skill 正文 <200 词、普通 <500 词量级——常驻元数据
  都是上下文税。

## 三、与本 skill 工作流的接线

- Task A 第 3 步（评测先行）：**纪律型 skill 加压力场景基线**——先裸跑记录
  借口原话，借口反驳表从真实基线来，不从想象来。
- Task B（修改）：改纪律型规则后，用原压力场景回归 + 新借口进反驳表。
- Task C（诊断·执行类）：agent 不守规 → 按防弹三件套补（封堵/精神条款/红旗单）。
- 写/改 description 时：先过 CSO 规则（无工作流摘要），再过指南四原则。
