# 写作（指南第三部分）

## §3.1 Level 1：Description（最重要的一层，写好等于成功一半）

**公式：功能定义 + 触发场景 + 核心能力（+ 排除边界，强烈建议）**

四原则：
1. 同时回答 WHAT 和 WHEN；
2. 枚举具体触发词，**包括口语化说法**（"建个AB"、"新建实验"都写进去）；
3. 第三人称（注入系统提示词的是客观描述，不是 "I can help you..."）；
4. 划排除边界——"写清不做什么"往往比"写清做什么"更有效。

```yaml
# ❌ 太笼统            description: 处理代码迁移
# ❌ 内部黑话          description: 处理 TCC 模块的 QCMS 迁移
# ✅ 完整：从哪到哪、适用条件、触发词、排除
description: >
  将项目中的旧版 HTTP 客户端迁移到新版统一请求库。适用于 Go 项目使用了
  old-http-client、需替换为 unified-httpclient 的场景。包含 import 替换、参数
  适配和错误处理改造。当用户说"迁移HTTP客户端"、"替换旧版请求库"时调用。
  不适用于其他语言或仅改配置的场景。
```

**20 问测试**：10 该触发 + 10 不该触发，命中率 <85% 回来调——该触发没触发→补
触发词；不该触发却触发→补排除边界。**中文团队建议 description 用中文或双语**。
多 skill 冲突：给区分关键词 / 显式写"本 skill 不处理 X 请用 yyy" / 常一起触发就
考虑合并。经验：竞品 description 若带全类目 MUST，仅靠加强自己可能赢不了——
按故障树给对方加排除才是正解（见 troubleshooting）。字数 ≤1024 字符。

## §3.2 Level 2 正文：六写法（各对一个 LLM 弱点）

1. **开头三件事**：做什么 / 为什么 / 何时不做（附具体检查命令让 Agent 快速判断）。
2. **祈使句 + 判断规则**：`检查 Go 版本，<1.18 → interface{}，>=1.18 → 原生泛型`
   ——不用"你需要/你应该"。
3. **解释为什么**：不堆 MUST。"字符串拼接会导致 SQL 注入——攻击者可输入 '; DROP
   TABLE'"。理解了才能举一反三。
4. **Before/After 对比**：diff 格式最直观，让 Agent 清楚"改什么、改成什么"。
5. **Few-Shot 3–5 个**：正常/边界/错误各一，输入输出成对，示例间有差异，
   **最典型的放第一个**（AI 模仿靠前的）。
6. **关键步骤加检查点**：`grep -rn "old-http-client" . | wc -l` 预期 0——让 Agent
   自我校验、及时纠偏。
7. **复杂逻辑下沉脚本**：正文只 `bash scripts/pre-check.sh`，脚本返回结构化 JSON。

**工作流加反馈闭环**（Plan → Validate → Execute）：校验失败要写明"回退到 Step N"，
不可逆操作前必有 Validate 节点。

## §3.3 防 LLM 偷懒的四武器

| 武器 | 用法 |
|---|---|
| 强硬语气 | "Delete it. Start over. 不要修改，不要妥协。"——命令式遵从率高于"你应该" |
| 借口反驳表 | 预判自我合理化并逐条堵死："代码看起来对了，跳过验证"→不行，"看起来对"和"测试证明对"之间隔着一堆 bug |
| 量化阈值 | "分析至少 10 个函数，每个 ≥3 个不变量、≥5 个假设"——硬数字替代"足够多" |
| 负面指令 | "Do NOT curl the deployed URL to verify——要看 dashboard 状态才算数" |

## §3.5 Level 3 支撑层（按需加载，放心放）

```
my-skill/
├── SKILL.md        ← L1 frontmatter + L2 正文（≤500 行）
├── references/     ← 长规范、API 文档、业务背景（>50 行的说明性内容下沉）
├── scripts/        ← 确定性计算（LLM 有概率错、脚本 100% 对的事）
├── examples/       ← 优质输出样例 1–3 个（可含一个标注错处的 bad-output）
└── evals/          ← 评测用例集
```

**脚本四原则**：自愈性（异常也正常退出、输出 `{"status":"error",...}`）/
结构化输出（统一 JSON）/ 幂等（重复执行结果一致）/ 安全边界（不越权、不硬编码
密钥）。适合封装：配置读写、环境检测、数值计算、HTTP 调用。
