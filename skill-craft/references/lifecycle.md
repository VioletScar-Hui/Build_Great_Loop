# 迭代与维护（指南第八部分：版本 / 泛化 / 合并拆分 / 退役）

## 版本控制

frontmatter 标版本号，按改动大小升：

| 变更类型 | 版本 | 示例 |
|---|---|---|
| 修复/补充说明 | 1.0 → 1.1 | 修某边界场景判断 |
| 新增步骤或场景 | 1.1 → 1.2 | 增加 gRPC 支持 |
| 重大结构调整/流程重写 | 1.x → 2.0 | 单体拆模块化 |

团队协作：像代码一样 Review（走 PR），维护 CHANGELOG 记录每次变更原因。

## 泛化 vs 特化

**起点选特化，真实 case 验证；同一规则多处重复时再泛化。**

| 信号 | 操作 |
|---|---|
| 同一规则在多个 skill 重复 | 泛化：提取通用 skill 或共享规范 |
| skill 越写越大、场景越加越多 | 特化：拆成更小粒度 |

## 合并 / 拆分操作要点

- **拆分**触发：见 decide-and-design §粒度信号。拆后主 SKILL.md 只留意图路由表 +
  全局红线；每个子模块引用写契约。
- **合并**触发：两个 skill 频繁一起触发 / 语义重叠且排除边界救不回 / 各自触发
  准确率 <80%。合并保留 description 更清晰的那个名字，另一个的触发词并入，
  evals 两边全迁移，合并后跑一轮回归 + 触发测试。
- 无论合并或拆分：**先归档原状**（副本移出 skills 目录），可回退。

## 退役条件

| 退役信号 | 说明 |
|---|---|
| 模型原生能力已覆盖 | 新模型直接能做对，不再需要约束 |
| 维护成本超收益 | 每次升级都要改，实际用得少 |
| 流程已过时 | 业务变了，场景不再发生 |
| 与新 skill 大量重叠 | 合并，保留更清晰的那个 |

**退役流程**：向用户复述该 skill 现状与退役理由 → 用户确认 → 归档（移出
skills 目录保留副本，含 evals）→ 检查是否有其他 skill 引用它（引用处改指向）。

> 退役不是失败。skill 太多互相打架，Agent 反而更难做对；每个常驻 description
> 都是上下文税。留下能被复用、能被测试、能被接手维护的那部分。
