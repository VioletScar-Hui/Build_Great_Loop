# 未知项引导（四象限引出法）

> 来源：Thariq Shihipar (Claude Code 工程师) "A Field Guide to Fable: Finding Your
> Unknowns"（2026-07）。核心：地图（提示词/spec）≠ 疆域（真实约束），差距=未知项；
> 质量瓶颈在澄清未知项的能力。访谈的本职就是系统化地消灭未知项——按象限选招。

## 四象限 → 各用什么招

| 象限 | 识别信号 | 引出手段 |
|---|---|---|
| 已知的已知 | 用户主动说出的 | 确认，不重问（画像/预填规则） |
| 已知的未知 | 用户说"还没想好 X" | 常规问题轮（2–4/轮、带选项、问题库为底） |
| **未知的已知** | "说不清，但看到就知道要不要" | **样例先行**（见下） |
| **未知的未知** | 用户对领域不熟 / 从没做过这类任务 | **盲点扫描**（见下） |

## 招式一：盲点扫描（blindspot pass）

**触发**：Phase 0 加一个便宜的熟悉度探测——"这个领域/这类任务你熟吗？"（或从上下文
推断）。用户不熟 → 在提问之前先做一轮盲点扫描：

- 列出这个领域的**经典陷阱、质量维度、行家才知道该问的问题**（3–6 条，每条一句
  为什么重要），让用户先"知道自己不知道什么"，再进入问题轮。
- 效果：把"未知的未知"降级为"已知的未知"，后续问题轮才问得到点子上。

**实证**：legal-agent-industry-loop Run 1 的 gotcha-1（转载源冒充独立源）正是漏掉的
盲点——"来源独立性如何判定"是研究领域的行家常识，用户不知道要问，问题库当时也没
预置。对不熟领域，盲点扫描是问题库之上的动态补充层。

## 招式二：样例先行（see-it-to-know-it）

**触发**：质量目标模糊、用户反应是"说不好，看到就知道"（简报长相、标注口径、文风、
版式……）。**不要让用户批准抽象标准**——先花一个便宜增量做 **2–4 版差异大的微样例**
（半屏即可：两种简报开头、三条不同口径的标注示例），让用户**挑和改**，再把反应反推
成 STANDARDS 条款。

- 时机：Phase 2 起草标准**之前**。规格上的小差异会导致实现的巨大差异，样例阶段发现
  远比实现后返工便宜。
- 纪律：样例是引出工具不是交付物，做完即弃；每版必须真有差异（不是同款微调）。

## 招式三：参考物优先（reference-first）

**最便宜的规格压缩：一个好参考物 > 几百字描述。** 每次访谈固定问一句：

> "有没有现成的参考物——往期满意的产出 / 喜欢的样式或网站 / 实现了类似行为的代码？"

有 → 让 agent 直接读它（**源码/原件优于截图和转述**），从中反推标准与口径，用户只
确认差异点。参考物是另一种语言/格式也没关系。

## 张力说明：具体 vs 灵活

访谈把 SPEC 钉得很细，会不会让 loop 死板执行、错过更好的路？分层解决：
- SPEC 钉死的是 **WHAT**（目标、标准、红线）——这些就该死板；
- **HOW** 留给循环内的计划迭代 + **Deviations 协议**（遇边界情况：选保守方案 → 记录
  原因 → 继续；见 loop-engineering harness 模板）——疆域给出的新信息在这里消化；
- 若疆域信息动摇的是 WHAT 本身（spec 的假设错了）→ human gate，不自作主张。
